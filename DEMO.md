# Demo Walkthrough

公开演示收敛为三条路径。它们不是功能清单，而是三段连续体验：先建立关系，再收敛注意力，再进入一个能被看见和修改的决策工作台。

## 1. 首次进入：用最少信息完成首次激活

![Onboarding Scene](./assets/scene-onboarding.svg)

### 想展示什么

- 这不是一个上来就索取大量资料的长表单
- assistant 先建立身份，再开始理解用户
- 首次建档只收长期稳定、解释成本低的信息

### 演示路径

用户进入产品后，首先看见的是一个具有明确身份的 assistant，而不是配置面板。流程依次收最小必要信息：称呼、地点、生活节奏，以及少量阶段标签。完成后，用户先进入可浏览的应用壳层；依赖模型的能力可以稍后开启。

### 公开 demo 的边界

- 不演示真实密钥输入
- 不演示真实模型调用
- 只演示“最小激活”这条关系建立路径

## 2. 今日承接：首页只接住最该处理的一件事

![Today Scene](./assets/scene-today.svg)

### 想展示什么

- 首页不是信息堆积，而是注意力收敛
- 页面优先承接已经足够明确的结果
- assistant 只在复杂修改、追问和跨场景协调时出现

### 演示路径

进入“今天”页后，用户会先看到当下最需要被接住的一件事，而不是一组平级卡片。主卡提供单步动作，例如继续、确认、查看详情；如果用户想换条件、补充约束或重新判断，再交给 assistant 继续处理。

### 公开 demo 的边界

- 日程和推荐内容使用静态展示数据
- 不暴露内部 page model 和 runtime mapping
- 重点展示信息架构和决策承接方式

## 3. 午餐工作台：从默认判断进入可修改的决策流

![Decision Scene](./assets/scene-decision.svg)

### 想展示什么

- 单个 skill 场景如何被做成一个完整的工作台
- 系统如何基于预算、地点、时间和偏好给出默认候选
- 用户如何继续追问、切换、删除或确认

### 演示路径

展示版以午餐事项为例。首页先给出当前建议，详情页再把候选、上下文和对话承接放到一起。默认方案不是唯一答案，它只是一个足够稳妥的起点。用户可以保留默认、继续修改，或把它交还给 assistant 继续处理。

### 公开 demo 的边界

- 候选数据来自本地模拟内容
- 不连接真实外部平台
- 不展示商家级或 provider 级真实接入信息

## 如何讲这套 demo

如果用于介绍项目，建议按以下顺序讲：

1. 先讲“为什么需要一个做生活决策的 AI 助理”
2. 再讲“为什么不是只做聊天，而是做承接和执行边界”
3. 最后用这三条路径证明：
   - 它能建立关系
   - 它能收敛注意力
   - 它能把单个决策场景做成可以继续推进的产品体验

## 演示方式建议

- 录屏/GIF：适合快速分享和嵌入 README
- 静态网页：适合快速浏览体验和对外演示
- 补充文档：用 `VISION.md` 和 `ARCHITECTURE.md` 承接“为什么这样做”
