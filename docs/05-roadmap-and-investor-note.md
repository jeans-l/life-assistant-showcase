# 05｜Roadmap & Investor Note：从 P0 Demo 到个人生活决策基础设施

> 本文件独立说明生活助理的阶段规划、验证指标、投资叙事和对外表达。目标是让外部读者理解：这不是一个小工具，而是一个有清晰起点和长期扩展空间的 AI-native 产品方向。

---

## 1. 一句话投资叙事

生活助理探索的是：

> AI 能否成为个人生活中的决策基础设施。

它不是另一个 chatbot，也不是一个日程工具，而是一个能基于 Memory、当前状态、可调用 Skill 和外部服务，替用户承担高频、低风险、重复性决策的个人操作层。

长期机会是：

> 让过去只有少数人通过助理、教练、顾问才能获得的持续判断能力，变成普通人也能拥有的 AI 基础设施。

---

## 2. 为什么现在值得做

### 2.1 供给过剩带来决策成本上升

今天用户不是没有选择，而是选择太多。

典型场景：

1. 吃什么：餐厅、外卖、健康、预算、时间都要考虑。
2. 买什么：价格、品牌、库存、复购周期都要判断。
3. 学什么：目标、时间、难度、反馈都要协调。
4. 看什么：信息太多，重要性排序困难。
5. 怎么安排一天：时间、精力、地点、任务相互影响。

信息越丰富，决策成本越高。

---

### 2.2 Chatbot 解决不了完整闭环

Chatbot 可以回答问题，但生活场景需要的是：

1. 记住我是谁。
2. 知道我今天怎样。
3. 主动接住当前事项。
4. 给出默认判断。
5. 低风险时推进执行。
6. 结果写回日程、记忆和页面。
7. 根据反馈修正下一次。

这要求系统从“对话”升级为“受治理的决策与执行 runtime”。

---

### 2.3 AI 能力正在从生成走向代理

第一阶段 AI 产品强调生成内容。

下一阶段更重要的是：

1. AI 如何理解用户长期状态。
2. AI 如何在具体上下文中行动。
3. AI 如何连接外部服务。
4. AI 如何判断什么可以自动做，什么必须确认。
5. AI 如何从反馈中持续学习。

生活助理正是围绕这五个问题设计。

---

## 3. 阶段规划

## P0：核心体验验证

### 目标

验证一个最小命题：

> 用户是否愿意让 AI 在高频、低风险、重复性的生活场景里给出默认判断，并通过轻量确认或反馈持续调整。

---

### 范围

P0 做：

1. NOW 当前事项。
2. 日程时间结构。
3. 角色 / Skill 管理。
4. 我的 / Memory 与设置。
5. 全局 assistant 半屏 / 全屏。
6. 饮食助手主链路。
7. 最近判断与反馈。
8. 本地 demo 数据和状态写回。

---

### 不做

P0 不做：

1. 真实外卖下单。
2. Skill 市场。
3. 远程 Skill 托管。
4. 高风险自动执行。
5. 常驻后台 Agent。
6. 多 Skill 自动协作面板。
7. 聊天内复杂卡片和表单。

---

### P0 Demo 必须展示

1. NOW 午餐默认决策。
2. 日程中通过 assistant 调整午餐时间。
3. 饮食助手设置和启用。
4. 最近判断解释和记忆修正。

---

### P0 成功标准

产品体验层：

1. 用户能在 3 分钟内理解产品不是普通日程或聊天工具。
2. 用户能看懂 NOW 是当前注意力收敛页。
3. 用户能理解 assistant 结果会写回页面。
4. 用户能理解 Skill 是能力模块，不是人格。
5. 用户能理解反馈会影响后续判断。

功能层：

1. 四个主页面可用。
2. assistant 半屏可从任意主页面打开。
3. 至少一条事项可从 NOW 到日程同步。
4. 至少一个 Skill 可完成设置和启用。
5. 至少一条最近判断可解释并反馈。

---

## P1：真实服务能力增强

### 目标

从“演示闭环”进入“可用闭环”。

重点是让核心场景不只是 mock，而是真正能接入外部服务和模型能力。

---

### 范围

P1 做：

1. 更完整的 MealSkill。
2. 外卖服务候选查询。
3. 模型服务配置。
4. NewsSkill 主动摘要。
5. LearningSkill 学习计划。
6. 服务状态管理。
7. 日程冲突检测。
8. 最近判断增强。
9. Memory 候选管理。

---

### 关键能力

#### MealSkill 增强

1. 真实候选拉取。
2. 预算、口味、配送时间排序。
3. 用户反馈收集。
4. 餐史和稳定偏好沉淀。

#### NewsSkill 增强

1. 主题偏好设置。
2. 每日摘要。
3. 重要信息高亮。
4. 阅读反馈。

#### LearningSkill 增强

1. 学习目标设置。
2. 日计划生成。
3. 完成情况反馈。
4. 中断后的恢复方案。

---

### P1 成功标准

1. 至少一个 Skill 能接入真实外部服务或真实数据源。
2. assistant 能根据服务状态降级。
3. 用户反馈能影响下一次同类判断。
4. 最近判断能解释关键依据。
5. 日程调整不再只是静态 demo。

---

## P2：可扩展 Assistant Platform

### 目标

从“生活助理产品”升级为“可扩展的个人 assistant 平台”。

核心是 Skill 与 Provider 的生态化。

---

### 范围

P2 做：

1. Skill marketplace。
2. Remote hosted skill。
3. Provider ecosystem。
4. Minimal control plane。
5. Skill manifest。
6. Cross-skill insight。
7. Advanced memory correction。
8. Personal automation workflows。

---

### 关键能力

#### Skill Marketplace

用户可以添加更多能力模块，例如：

1. 买什么。
2. 健身助手。
3. 旅行计划。
4. 家庭补货。
5. 合同初筛。
6. 个人财务提醒。

---

#### Remote Hosted Skill

外部服务商可以提供自己的 Skill 能力，但必须遵守：

1. 通过 manifest 注册。
2. 只接收 governed context。
3. 不拿用户真源数据句柄。
4. 不持有 provider token。
5. 不拥有写回主权。

---

#### Provider Ecosystem

Provider 提供外部候选、执行和状态回流。

例如：

1. 外卖平台。
2. 电商平台。
3. 日历服务。
4. 地图服务。
5. 天气服务。
6. 学习内容平台。

Provider 不是智能主体，只是外部能力来源。

## 4. 验证指标

### 4.1 总体验证指标

| 指标 | 目标 | 含义 |
|---|---|---|
| 默认方案接受率 | >= 60% 起步，逐步提升 | 用户是否愿意接受 AI 默认判断 |
| 调整后采纳率 | >= 70% | 用户轻量调整后是否仍愿意继续 |
| 反馈闭环完成率 | >= 50% 起步 | 用户是否愿意提供学习信号 |
| 次日/次周复用率 | 持续观察 | 系统是否形成连续服务 |
| 用户主动唤起 assistant 比例 | 持续观察 | assistant 是否成为自然入口 |

---

### 4.2 MealSkill 指标

| 指标 | 目标 | 含义 |
|---|---|---|
| 午餐默认建议接受率 | >= 70% | 高频低风险场景是否成立 |
| 持续拒绝率 | <= 20% | 默认判断是否偏离用户 |
| 调整原因收集率 | >= 50% | 是否能获得学习信号 |
| 同类问题重复改善 | 可观察 | 系统是否真的变好 |

---

### 4.3 NewsSkill 指标

| 指标 | 目标 | 含义 |
|---|---|---|
| 摘要打开率 | >= 50% | 用户是否需要主动摘要 |
| 阅读完成率 | >= 40% | 内容质量是否足够 |
| 正向反馈率 | >= 20% | 主题偏好是否匹配 |

---

### 4.4 LearningSkill 指标

| 指标 | 目标 | 含义 |
|---|---|---|
| 任务完成率 | >= 50% 起步 | 学习计划是否可执行 |
| 中断恢复率 | 可观察 | assistant 是否能降低放弃率 |
| 调整后继续率 | >= 60% | 计划是否能适应真实生活 |

## 5. 为什么从“吃什么”切入

“吃什么”是一个很好的 MVP 场景，因为它同时具备：

1. 高频：每天都会发生。
2. 低风险：决策错误成本相对低。
3. 有偏好：口味、预算、健康、时间都个性化。
4. 有外部服务：外卖、餐厅、配送可接入。
5. 有反馈：好吃、不好吃、太油、太冷、太贵。
6. 可执行：从建议到下单存在清晰路径。
7. 可学习：反馈会明显影响下一次判断。

它能验证生活助理最核心的问题：

> 用户是否愿意让 AI 替自己做一个日常默认决策。

---

## 6. 产品护城河

生活助理的护城河不是“用了某个模型”，而是以下组合：

### 6.1 个人 Memory

长期偏好、当天状态、本轮约束、历史反馈持续积累，形成个人化判断基础。

### 6.2 Item 模型

把生活中的事情抽象成可推进、可执行、可写回的事项，而不是停留在聊天消息或日程事件。

### 6.3 Runtime Harness

多入口统一进入同一条受治理主链，保证系统可扩展、可解释、可控。

### 6.4 Skill 生态

不同领域能力可以作为 Skill 接入，但不破坏 assistant 主权。

### 6.5 Writeback Policy

系统知道什么能写、写到哪、何时需要确认、何时只作为候选。

---

## 7. 风险与应对

### 7.1 用户不愿意被主动打扰

应对：

1. NOW 只展示当前一件事。
2. 主动性可设置。
3. 低风险才主动推进。
4. 关键动作保留确认。
5. 用户可以关闭 Skill。

---

### 7.2 默认判断不够准

应对：

1. 先做低风险高频场景。
2. 默认方案可快速调整。
3. 反馈进入候选，不直接污染长期记忆。
4. 多次重复后再提升。

---

### 7.3 外部服务接入复杂

应对：

1. P0 先 mock，不伪装真实执行。
2. P1 接入最小候选查询。
3. 高风险执行必须确认。
4. Provider 与 Skill 分离，避免耦合。

---

### 7.4 产品容易变成“大而全”

应对：

1. P0 只验证午餐主链。
2. P1 再扩展 News 和 Learning。
3. P2 才做 Skill 生态。
4. 每个阶段都有明确不做清单。

## 11. 一句话总结

生活助理的路线不是先做全能 AI，而是：

> 从一个高频、低风险、可反馈的生活决策开始，验证用户是否愿意把默认判断交给 AI；再用 Memory、Skill、Provider 和 Writeback 机制，把这个闭环扩展成个人生活的决策基础设施。
