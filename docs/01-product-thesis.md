# 01｜产品思路：从聊天工具到生活决策 Harness

> 本文件独立说明生活助理的产品思路：为什么它不是普通聊天 App、不是普通日程 App，而是一个围绕 Memory、Item、Skill、Provider 和 Writeback 的 AI 生活决策 Harness。

---

## 1. 产品定位

生活助理的核心定位是：

> 一个基于 Memory 的个人 AI 决策助手。

它不是单纯回答问题，而是在用户尽量少重复输入的前提下，持续理解用户目标、偏好和边界，在合适的时间给出可执行的默认方案，并在必要时帮助用户调整。

MVP 要验证的核心不是“聊天更像人”，而是：

> 用户是否愿意把高频、低风险、重复性的生活决策委托给 AI 助理。

---

## 2. 为什么不是普通 Chatbot

普通 Chatbot 的基本链路是：

```text
用户提问 → AI 回答 → 对话结束
```

这条链路的问题是：

1. 用户每次都要主动提问。
2. AI 每次都依赖当前对话，难以稳定复用长期偏好。
3. 回答通常停留在文字里，不进入日程、记忆或外部执行。
4. 反馈很难成为下一次判断的正式依据。
5. 对生活场景来说，用户要的往往不是解释，而是“帮我接住这件事”。

生活助理的链路是：

```text
触发 → 读取上下文 → 接住事项 → 调用能力 → 形成判断 → 执行或等待确认 → 写回 → 学习
```

Chat 在这里仍然重要，但它只是入口之一。

---

## 3. 为什么不是普通日程 App

普通日程 App 的核心对象是 event：

```text
某天某时有一件事
```

生活助理的核心对象是 item：

```text
一件需要被理解、推进、调整、执行和反馈的生活事项
```

两者差异：

| 维度 | 普通日程 Event | 生活助理 Item |
|---|---|---|
| 核心目标 | 记录时间 | 推进一件事 |
| 上下文 | 标题、时间、地点 | 目标、约束、状态、动作、反馈 |
| AI 作用 | 辅助创建或编辑 | 判断、协调、执行、写回 |
| 变化处理 | 用户手动改 | assistant 判断影响并回写 |
| 学习机制 | 基本没有 | 反馈进入 memory 和 decision records |

例如“午餐”不是一个简单 11:30-13:00 的事件，而是：

1. 有目标：解决今天午餐。
2. 有时间窗：11:30-13:00。
3. 有状态：准备中、进行中、已完成。
4. 有偏好：清淡、高蛋白、预算 50 元内。
5. 有结果：轻食鸡胸能量碗。
6. 有外部服务：外卖候选和配送状态。
7. 有反馈：以后别太冷、别太晚、别太油。

日程页只是 item 在时间结构中的投影，不是系统的全部。

---

## 4. 为什么需要 Harness 思想

这里的 Harness 不是一个视觉概念，而是一种产品与系统组织方式。

生活助理需要把不同来源的输入、能力和结果收敛到一个受治理的运行闭环里：

```text
用户输入
页面操作
时间窗口
外部服务
历史反馈
长期偏好
当天状态
```

这些东西不能各自为政。它们需要进入同一条链路：

```text
Entry
→ Route
→ Context Load
→ Domain Judgment
→ Validation
→ State Recheck
→ Policy Decision
→ Execution or Handoff
→ Writeback Decision
→ Return Sync
→ Correction
```

产品上体现为：

1. 用户可以从任意页面唤起 assistant。
2. assistant 能带入当前页面上下文。
3. Skill 被调用前有统一路由和上下文装配。
4. Skill 结果不能直接写页面。
5. 写回必须经过统一裁决。
6. 页面只消费已裁决结果并刷新。

这就是生活助理与普通“AI + 页面”的区别。

---

## 5. 五个产品判断

### 5.1 用户需要默认决策，而不是更多候选

大量生活决策不是因为用户想比较，而是因为用户不想再想。

午餐、日用品、学习安排、提醒时间、出门准备这类场景，用户真正需要的是：

1. 系统理解我的偏好。
2. 系统给一个默认方案。
3. 我觉得不对时能快速调整。
4. 系统记住这次调整。

所以生活助理默认不应该给一堆列表，而应该先给一个可解释、可调整、可反馈的默认方案。

---

### 5.2 用户保留目标定义权，assistant 承担过程判断权

生活助理不是替用户定义人生目标，而是在用户给定目标和边界后，承担重复性过程判断。

分工是：

| 用户 | Assistant |
|---|---|
| 定义目标 | 理解目标 |
| 确定价值取向 | 提取约束 |
| 授权高风险动作 | 做低风险默认判断 |
| 纠正不满意结果 | 记住反馈并调整 |

例如用户定义“最近想吃健康点”，assistant 可以在午餐窗口更偏向清淡、高蛋白、少油的方案。但如果涉及高金额购买、健康风险、重要会议调整，assistant 必须等待用户确认。

---

### 5.3 页面负责承接，assistant 负责判断

页面不是智能主体。

页面负责：

1. 展示结果。
2. 让用户查看详情。
3. 支持明确字段编辑。
4. 承接保存、删除、确认。
5. 展示 assistant 返回后的同步结果。

assistant 负责：

1. 判断是否需要做。
2. 判断调用哪个 Skill。
3. 判断是否需要外部服务。
4. 判断是否需要用户确认。
5. 判断结果是否能写回。
6. 判断反馈应该进入哪层 memory。

这能避免页面逻辑越来越复杂，也能保证 AI 能力有统一治理边界。

---

### 5.4 Skill 是可治理能力，不是独立人格

Skill 是领域能力包，负责形成领域结果。

例如：

| Skill | 负责什么 |
|---|---|
| 饮食助手 | 餐食偏好、候选、预算、配送、口味 |
| News 简报 | 信息主题、摘要、重要性排序 |
| 学习助手 | 学习目标、计划、回顾、节奏调整 |

但 Skill 不直接决定：

1. 是否打扰用户。
2. 是否执行外部动作。
3. 是否写入日程。
4. 是否修改长期记忆。
5. 是否同步到 NOW。

这些都由 assistant 统一处理。

---

### 5.5 反馈不是点赞，而是学习入口

用户反馈不应只是“喜欢 / 不喜欢”。

反馈至少可能影响三类东西：

1. **本轮约束**：这次别吃冷的。
2. **当天状态**：今天想吃热一点。
3. **长期候选**：午餐长期不喜欢冷食。

系统不能把一次反馈直接写成长期人设，也不能丢掉反馈。正确方式是：

```text
反馈
→ 结构化候选
→ 写回裁决
→ hold / promote / rollback / expire
→ 后续判断读取
```

这能让系统稳步变聪明，同时避免误学。

## 6. 核心产品对象

### 6.1 Memory

Memory 不是一个统一记忆池，而是分层上下文系统。

至少包括：

| 层级 | 作用 | 示例 |
|---|---|---|
| profile | 长期稳定默认 | 常驻位置、预算、长期偏好 |
| assistantSoul | 助手风格 | 简洁、直接、少打扰 |
| today | 今天状态 | 天气、地点、精力、当天安排 |
| conversationContext | 本轮约束 | 这次想吃热一点 |
| skill-local | 领域偏好 | 饮食预算、忌口、学习节奏 |
| decisionRecords | 判断依据 | 当时为什么这么推荐 |
| crossSkillInsights | 跨场景规律 | 低精力日减少高强度安排 |

Memory 的原则是：按需读取、分层写回、证据升级。

---

### 6.2 Item

Item 是 assistant 正式接住的一件事。

最小构成：

1. goal：这件事要推进什么。
2. window：它在哪个时间窗发生。
3. lifecycle：它当前处于哪个阶段。
4. relatedActions：它有哪些准备、过渡、收尾动作。
5. ownership：谁负责最后推进。
6. cadence：一次性、重复性或条件触发。
7. presentationContract：前台如何承接。

---

### 6.3 Skill

Skill 是领域能力模块。

它输出的是 proposal、domain state、domain actions、writeback candidate，而不是最终页面状态。

---

### 6.4 Provider

Provider 是外部能力来源。

它可以提供：

1. 外部候选。
2. 外部执行。
3. 状态回流。
4. 授权状态。

Provider 不是智能主体，不决定是否采纳结果。

---

### 6.5 Writeback

Writeback 是系统真正进入生活的关键。

没有写回，AI 只是回答；有受治理写回，AI 才能成为助理。

写回对象包括：

1. item。
2. memory。
3. today state。
4. decisionRecords。
5. projection。
6. page sync。

---

## 7. MVP 验证命题

MVP 只验证一个核心命题：

> 用户是否愿意把高频、低风险、重复性的生活决策委托给 AI 助理。

拆成三个子命题：

### 7.1 产品命题

用户愿不愿意接受系统给出的默认方案，而不是每次自己从零选择。

### 7.2 行为命题

如果系统利用长期档案、当天状态和历史反馈给出默认方案，用户是否更愿意接受并轻量调整。

### 7.3 能力命题

系统是否能在反馈中持续学习，让下一次判断更稳定。

## 8. 典型用户体验

### 8.1 午餐

```text
11:20
用户打开 NOW
看到午餐卡片
系统已经给出一个轻食方案
用户点确认
结果写回今天事项
后续如果用户反馈“太冷”
系统把它作为候选偏好
```

### 8.2 日程调整

```text
用户在日程看到午餐 11:30-13:00
输入“帮我把午餐晚一点”
assistant 半屏打开
它知道当前页面、当前事件和后续安排
给出 12:30 或 13:00
用户选择后写回日程
```

### 8.3 Skill 设置

```text
用户进入角色页
看到饮食助手待设置
设置口味、预算、忌口、常用时间
启用后 assistant 才能调用它参与午餐判断
```

### 8.4 最近判断

```text
用户进入我的
查看最近判断
看到系统为什么推荐轻食
用户纠正“以后午餐别太冷”
系统记录为候选，不立即污染长期偏好
```

---

## 9. 产品边界

生活助理不会在 P0 做：

1. 高风险自动执行。
2. 无确认的真实下单。
3. 三方 Skill 市场。
4. 完整后台常驻 Agent。
5. 聊天内复杂工作流。
6. 全场景生活覆盖。

P0 只做：

1. 一个清楚的 App 舞台。
2. 一条可感知的 assistant 主链。
3. 一个可展示的饮食决策闭环。
4. 一套可理解的 Skill 管理。
5. 一套可解释和可纠正的 memory 机制。

---

## 10. 一句话总结

生活助理的产品思路是：

> 用 Memory 理解用户，用 Item 接住生活事项，用 Skill 提供领域能力，用 Provider 连接外部服务，用 Writeback 把 AI 判断变成真实生活状态。
