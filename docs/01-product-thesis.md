# 01｜Product Thesis

> 这份文档用于说明生活助理的主要产品思路：它不是普通日程 App，也不是普通 AI Chat，而是一个面向个人生活决策的 AI harness。

## 1. 核心判断

AI App 的下一阶段，不应只停留在“对话生成”。

对话当然重要，但如果 AI 只是在聊天窗口里给建议，用户仍然需要自己判断、自己切换工具、自己执行、自己记住结果。这样的产品本质上还是“更聪明的信息工具”。

生活助理要验证的是另一件事：

> AI 是否可以在用户授权和可控边界内，替用户承担一部分高频、低风险、重复性的日常决策，并把结果稳定写回用户的生活结构。

这意味着产品核心不是 Chat，而是：

```text
目标 → 约束 → 决策 → 执行 → 反馈 → 学习
```

## 2. 不做什么

这个产品不是：

1. 不是普通日程工具：日程只是时间承接面，不是智能主体。
2. 不是普通聊天工具：Chat 是入口，不是最终结果停留地。
3. 不是推荐列表：目标不是给更多选择，而是给一个可信默认方案。
4. 不是一堆拟人 Agent：Skill 是能力模块，不是互相抢主权的角色。
5. 不是自动化脚本：执行前需要治理、确认、降级和写回裁决。

## 3. 产品主语：Item / 事项

生活助理正式接住的对象不是消息，而是 `Item / 事项`。

一件事项至少包含：

```text
目标
时间窗口
当前阶段
关联动作
执行边界
节奏特征
前台承接方式
```

例如“午餐”不是一句“吃什么”的问题，而是一件带有时间窗、偏好、预算、候选方案、确认动作、外部服务和反馈学习的事项。

## 4. 页面分工

移动端 App 是 assistant 的服务舞台，不是第二个智能主体。

```text
NOW：当前注意力收敛页，回答现在最该关注什么
日程：完整时间组织页，回答今天和接下来怎么排
角色：Skill 管理页，回答哪些能力可用、哪些需要设置
我的：长期默认、偏好、记忆、依据和控制权中心
全局助理：跨页面建议、判断、执行和写回层
```

全局助理不是第五个 Tab，而是所有页面共享的输入和对话层。

## 5. Assistant / Skill / Provider / Page 的边界

```text
Assistant：判断、协调、治理、写回
Skill：提供领域能力和领域结果
Provider：提供外部候选、执行和状态回流
Page：展示、编辑、确认和承接
Memory：提供长期默认、当天状态、当前上下文和历史证据
Item：承接用户生活中的一件正式事项
```

最关键的边界是：

> Skill 可以提出方案，但不直接写页面；Provider 可以执行动作，但不拥有判断主权；页面可以承接结果，但不决定长期记忆；正式写回必须由 Assistant Runtime 裁决。

## 6. 为什么强调 Writeback

AI 生活助理的难点不是生成一句看似合理的话，而是判断这句话能不能进入真实生活状态。

系统必须区分：

```text
本轮约束
今天状态
长期偏好
领域偏好
候选记忆
历史证据
页面同步
```

用户说“今天想吃热一点”，可能只影响本轮，也可能影响今天剩余时间，但不能直接升级成长期偏好。

用户连续三次在工作日午餐拒绝冷食，才可能形成一个候选规律。

所以生活助理的产品深度在于：

```text
candidate → writeback decision → hold / promote / rollback / expire → return sync
```

## 7. Harness 核心思想

这里的 harness 可以理解为一套受治理的运行框架：它把不同入口、不同能力、不同数据层和不同页面承接方式，收敛到同一条主链。

它解决四个问题：

1. 多入口统一：用户输入、页面操作、时间窗口、外部事件不割裂。
2. 多能力统一：Assistant Tool、Skill、Provider 都被 assistant 调用和裁决。
3. 多数据层统一：Memory、Item、Schedule、Decision Records 分层读写。
4. 多页面统一：NOW、日程、角色、我的都只消费已裁决结果。

## 8. 最小验证命题

P0 不验证“能否做全场景智能体”。

P0 只验证：

> 在午餐这类高频、低风险、重复性场景中，用户是否愿意接受 AI 给出的默认方案，并通过少量反馈让系统持续变好。

如果这个命题成立，再扩展到 News、Learning、买什么、健身、出行和更多生活服务场景。

## 9. 一句话总结

生活助理不是把 Chat 放进生活 App，而是把生活里的目标、约束、决策、执行、反馈和学习，放进一条统一的 Assistant Runtime。
