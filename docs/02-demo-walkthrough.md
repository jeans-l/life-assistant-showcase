# 02｜Demo Walkthrough：用四条链路展示核心闭环

> 本文件独立说明 GitHub 展示项目中的 Demo 应该做什么、做到什么程度，以及每条链路要证明哪个产品判断。

---

## 1. Demo 目标

P0 Demo 不追求全功能，不追求真实外部服务接入，不追求复杂自动化。

它只需要让用户、开发者和投资人看懂一件事：

> 生活助理不是聊天工具，也不是日程工具，而是能把 Memory、当前事项、Skill、外部服务和页面写回连接起来的 AI 决策与执行闭环。

Demo 的核心体验必须做到：

1. 用户能看到当前事项。
2. 系统能给出默认判断。
3. 用户能低成本确认或调整。
4. assistant 能带入页面上下文。
5. 结果能写回页面，而不是停留在聊天中。
6. 用户反馈能进入后续学习链路。

---

## 2. Demo 总原则

### 2.1 少做功能，多做闭环

不要堆页面和设置项。P0 Demo 只展示最能说明产品思想的 4 条链路：

1. NOW 默认决策。
2. 日程中通过 assistant 调整。
3. Skill 设置与启用。
4. 最近判断与记忆修正。

---

### 2.2 用 mock 数据，不伪装真实执行

Demo 可以使用 mock 数据，但必须区分：

| 类型 | 是否允许 | 展示方式 |
|---|---|---|
| Mock 餐食方案 | 允许 | 明确是 demo 数据 |
| Mock 外卖配送时间 | 允许 | 展示为样例候选 |
| 真实外卖下单 | P0 不做 | 不显示已真实下单 |
| 模型服务未接入 | 允许 | 明确提示能力受限 |
| 页面写回 | 必须做 | 本地状态更新即可 |

原则是：

> 可以演示体验，但不能伪装成已经接入真实服务。

---

### 2.3 Chat 只承接判断，不承接全部 UI

P0 对话层只做：

1. 纯文本消息。
2. 执行中状态。
3. 2-3 个快捷回复。
4. 结果摘要。
5. 失败或阻断说明。

不要在聊天里做复杂卡片、完整表单、多步骤审批。最终结果应回到 NOW、日程、角色或我的页面承接。

---

## 3. Demo 1：NOW 默认决策

### 3.1 目标

展示生活助理不是等用户提问，而是在合适的时间把当前最该关注的一件事收敛出来，并给出默认方案。

---

### 3.2 用户链路

```text
打开 App
→ 进入 NOW
→ 顶部显示日期、地点、天气
→ 当前事项卡片显示“午餐”
→ 系统给出默认午餐方案
→ 用户点击“确认”
→ 主卡状态更新为“已确认”
→ 日程里的午餐事件同步更新
```

---

### 3.3 页面状态

#### 初始状态

NOW 页面展示：

1. 日期：2026 年 04 月 27 日。
2. 星期：周一。
3. 地点：上海静安寺。
4. 天气：22°C。
5. 当前事项：午餐。
6. 时间窗：11:30 - 13:00。
7. 状态：准备中。
8. 默认方案：轻食鸡胸能量碗。
9. 价格：¥32。
10. 预计送达：约 18 分钟。

主按钮：`确认`。  
次按钮：`调整`。

快捷动作：

1. 换个更轻的。
2. 更快送达。
3. 晚点提醒我。

---

#### 用户点击“确认”后

页面更新：

1. 状态从 `准备中` 变为 `已确认`。
2. 主按钮变为 `已确认` 或收起。
3. 当前事项仍保留在 NOW，直到进入执行中或完成。
4. 日程页中的午餐事件同步展示已确认状态。
5. 最近判断中新增一条 demo 记录。

---

### 3.4 要证明的产品点

1. NOW 是注意力收敛页。
2. assistant 能基于上下文给默认判断。
3. 用户不需要从候选列表开始筛选。
4. 结果能写回页面。
5. 当前事项和领域方案是分开的：午餐是事项，轻食碗是方案。

---

### 3.5 实现程度

P0 必须做：

1. NOW 静态首屏。
2. 午餐主卡。
3. 确认按钮状态变化。
4. 本地状态写回。
5. 日程页事件块同步更新。

P0 可选做：

1. 点击餐食方案进入详情。
2. mock 执行中状态。
3. 简单 toast。

P0 不做：

1. 真实外卖下单。
2. 真实价格查询。
3. 真实配送状态。

## 4. Demo 2：日程中通过 Assistant 调整

### 4.1 目标

展示 assistant 能理解来源页面和当前事项，处理自然语言调整，并把结果同步回日程。

---

### 4.2 用户链路

```text
进入日程页
→ 看到午餐事件：11:30 - 13:00
→ 点击事件块进入详情
→ 点击“调整”或输入“帮我把午餐晚一点”
→ assistant 半屏打开
→ 上下文条显示：当前页面：日程；已带入：午餐 · 11:30-13:00
→ assistant 回复：我先看下后面安排
→ 执行中状态：正在检查今天的日程...
→ assistant 给出：12:30 / 13:00 / 先不改
→ 用户选择“改到 12:30”
→ assistant 返回结果摘要
→ 日程页事件块更新时间
```

---

### 4.3 半屏对话结构

半屏 assistant 需要展示：

1. 背景页面虚化或暗化。
2. 底部弹层。
3. assistant 头像。
4. 标题：assistant。
5. 副标题：基于当前日程页面。
6. 上下文条：当前页面 + 当前事件。
7. 用户消息。
8. assistant 消息。
9. 执行中状态。
10. 快捷回复按钮。
11. 底部输入框。

---

### 4.4 对话示例

```text
用户：帮我把午餐晚一点

assistant：可以，我先看下 12:30 后是否会影响后面的安排。

状态：正在检查今天的日程...

assistant：看起来 12:30 和 13:00 都可以，不会影响下午安排。你想改到哪个时间？

快捷按钮：
[改到 12:30] [改到 13:00] [先不改了]
```

用户点击 `改到 12:30` 后：

```text
assistant：已把午餐调整到 12:30 - 13:30，并同步到今天的日程。
```

---

### 4.5 日程写回效果

日程页更新：

1. 原午餐事件块从 11:30 - 13:00 移动到 12:30 - 13:30。
2. 事件详情中的时间同步更新。
3. NOW 如果仍以午餐为当前焦点，也同步显示新时间。
4. 最近判断记录本次调整原因。

---

### 4.6 要证明的产品点

1. assistant 不是孤立聊天，而是基于来源页面工作。
2. 快捷动作等价于向 assistant 发送带上下文的消息。
3. 页面不直接做复杂判断，assistant 判断后写回。
4. Return Sync 是体验闭环的关键。

---

### 4.7 实现程度

P0 必须做：

1. 日程主视图。
2. 事件块点击。
3. assistant 半屏。
4. mock 检查状态。
5. 快捷回复。
6. 本地事件时间更新。

P0 可选做：

1. 冲突提示。
2. 全屏 assistant。
3. 事件详情页完整展示。

P0 不做：

1. 拖拽改时间。
2. 复杂多事件重排。
3. 外部日历双向同步。

---

## 5. Demo 3：Skill 设置与启用

### 5.1 目标

展示 Skill 是可治理能力模块，不是默认自动运行的黑箱。

---

### 5.2 用户链路

```text
进入角色页
→ 看到三个内置 Skill
→ 饮食助手状态为待设置
→ 点击“去设置”
→ 进入饮食助手设置页
→ 设置口味偏好、预算范围、忌口、常用时间
→ 打开启用开关
→ 点击保存设置
→ 回到角色页
→ 饮食助手状态变为已启用
```

---

### 5.3 角色页展示

P0 展示三个 Skill：

| Skill | 初始状态 | 主按钮 | 说明 |
|---|---|---|---|
| 饮食助手 | 待设置 | 去设置 | 帮你安排每日三餐 |
| News 简报 | 待设置 | 去设置 | 帮你整理每天重要信息 |
| 学习助手 | 已停用 / 待设置 | 启用 / 去设置 | 帮你安排学习计划和回顾 |

原则：

1. 初始不默认启用任何 Skill。
2. 没有启用 Skill 不影响基础使用。
3. Skill 状态必须清晰可见。
4. Skill 详情页和设置页分开。

---

### 5.4 饮食助手设置页字段

P0 至少包含：

1. 启用 / 停用开关。
2. 口味偏好：清淡、米饭、热食、面食。
3. 预算范围：例如 20-50 元。
4. 忌口：少辣、不太油。
5. 常用时间：11:30-13:00。
6. 个性化备注：工作日午餐尽量轻一点，别太油。

---

### 5.5 保存后效果

保存成功后：

1. 饮食助手状态变为已启用。
2. 角色页卡片状态更新。
3. 饮食助手偏好进入 skill-local settings。
4. 后续 NOW 午餐卡片可以使用饮食助手提供领域方案。
5. 最近判断中可以说明“本次午餐建议参考了饮食助手设置”。

---

### 5.6 要证明的产品点

1. Skill 是用户可控能力。
2. assistant 调用 Skill 前需要用户授权或启用。
3. Skill 偏好和全局偏好分离。
4. Skill 不直接写页面，最终仍由 assistant 采用和写回。
5. 这套结构为后续 Skill 生态预留扩展空间。

---

### 5.7 实现程度

P0 必须做：

1. 角色页三个 Skill 卡片。
2. 饮食助手设置页。
3. 保存后状态变化。
4. 偏好本地存储。

P0 可选做：

1. Skill 详情页完整内容。
2. News / 学习助手设置页简化版。

P0 不做：

1. Skill 市场。
2. 发现更多角色。
3. 第三方 Skill 安装。

---

## 6. Demo 4：最近判断与记忆修正

### 6.1 目标

展示 AI 判断不是黑箱，用户可以查看依据、纠正结果，反馈会进入后续学习链路。

---

### 6.2 用户链路

```text
进入我的
→ 点击最近判断
→ 查看“为什么今天推荐轻食鸡胸能量碗”
→ 页面展示输入条件、使用偏好、最终建议、用户是否采纳
→ 用户点击“不太准确”或输入“以后午餐别太冷”
→ assistant 记录为候选偏好
→ 页面提示：已记录为候选，后续会结合更多反馈调整
```

---

### 6.3 最近判断展示字段

最近判断详情至少展示：

1. 判断时间：今天 11:20。
2. 关联事项：午餐。
3. 输入条件：午餐窗口、当前位置、预算、口味偏好。
4. 使用偏好：清淡、高蛋白、预算 50 元内。
5. 当天状态：下午有会议，适合轻一点。
6. 最终建议：轻食鸡胸能量碗。
7. 用户动作：已确认 / 已调整 / 已拒绝。
8. 反馈入口：不准确、以后别这样、补充偏好。

---

### 6.4 反馈后的写回规则

用户反馈“以后午餐别太冷”时，P0 不直接改写长期偏好。

正确处理是：

```text
用户反馈
→ 生成候选偏好：午餐不偏好冷食
→ 写入 decisionRecords
→ 可同步到 skill-local 候选
→ 后续重复出现后再提升为稳定偏好
```

页面文案建议：

```text
已记录：午餐尽量别太冷。
我会先作为候选偏好使用，后续如果你多次这样反馈，会把它提升为稳定偏好。
```

---

### 6.5 要证明的产品点

1. AI 判断可解释。
2. 用户能纠正系统。
3. 反馈不是简单点赞，而是进入 memory。
4. 系统不会因为一次反馈污染长期偏好。
5. 生活助理有持续学习机制。

---

### 6.6 实现程度

P0 必须做：

1. 我的页。
2. 最近判断入口。
3. 一条 mock 最近判断详情。
4. 反馈输入。
5. 候选偏好记录状态。

P0 可选做：

1. 最近聊天列表。
2. 记忆与习惯页。
3. 候选偏好管理。

P0 不做：

1. 完整长期记忆升级算法。
2. 复杂证据聚类。
3. 多天自动纠偏。

## 8. Demo 验收标准

Demo 完成后，用户应该能在 3 分钟内理解：

1. 这是一个 AI 个人生活助理，不是普通日程 App。
2. NOW 负责当前注意力收敛。
3. 日程负责完整时间结构。
4. 角色页负责 Skill 管理。
5. 我的页负责偏好、记忆和判断依据。
6. assistant 能从页面带上下文处理请求。
7. 结果会写回页面。
8. 用户反馈会进入后续学习。

如果用户看完只觉得“这是一个带聊天框的日程 App”，Demo 就失败了。

---

## 9. 一句话总结

P0 Demo 的目的不是展示功能有多全，而是证明：

> AI 助理可以基于 Memory 和当前事项给出默认判断，通过 Skill 和 Provider 获得能力，并在用户确认或反馈后把结果写回真实生活界面。
