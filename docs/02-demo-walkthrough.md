# 02｜Demo Walkthrough

> 这份文档定义 GitHub showcase demo 应该展示哪些主链路、做到什么程度，以及哪些内容暂时不做。

## 1. Demo 目标

Demo 的目标不是证明功能完整，而是让用户、投资人和开发者快速理解这个产品的核心闭环：

```text
当前事项 → 默认决策 → 用户确认 / 调整 → 写回页面 → 反馈学习
```

P0 Demo 应该让人 3 分钟看懂产品，10 分钟理解闭环，30 分钟看出底层架构思路。

## 2. Demo 范围

P0 Demo 只做 4 条主链路：

1. NOW 当前事项默认决策
2. 日程页通过 assistant 调整事件
3. 角色页设置并启用 Skill
4. 我的页查看最近判断与记忆反馈

不做全功能，不做真实外部履约，不做 Skill 市场。

---

## 3. Demo 1：NOW 当前事项默认决策

### 目标

展示生活助理不是等用户问，而是在合适的时间，把当前最该关注的一件事收敛出来，并给出默认方案。

### 链路

```text
打开 NOW
→ 顶部展示日期、地点、天气
→ 当前事项主卡展示“午餐”
→ 系统给出默认午餐方案
→ 用户点击“确认”
→ 主卡状态更新为已确认 / 已安排
```

### 页面状态

1. 初始状态：午餐准备中
2. 推荐方案：轻食鸡胸能量碗
3. 主按钮：确认
4. 次按钮：调整
5. 快捷动作：换个更轻的 / 更快送达 / 晚点提醒我

### 展示重点

1. NOW 是注意力收敛，不是信息流。
2. 系统给一个默认方案，不是推荐列表。
3. 用户可以低成本接受或调整。
4. 确认后结果写回页面，不停留在聊天里。

### 实现程度

```text
必须可交互：确认按钮、调整按钮
可以 mock：餐食图片、价格、配送时间
不用真实接入：外卖平台、支付、真实下单
```

## 4. Demo 2：日程页通过 assistant 调整事件

### 目标

展示 assistant 能从当前页面带入上下文，并完成一个有判断、有确认、有写回的修改链路。

### 链路

```text
进入日程页
→ 点击午餐事件
→ 输入“帮我把午餐晚一点”
→ assistant 半屏打开
→ 上下文条展示：当前页面 日程 / 已带入 午餐 11:30-13:00
→ assistant 判断 12:30 和 13:00 都不影响下午安排
→ 用户选择“改到 12:30”
→ 返回日程页
→ 午餐事件块时间更新为 12:30-13:30
```

### 半屏对话内容

```text
用户：帮我把午餐晚一点
assistant：可以，我先看下 12:30 后是否会影响后面的安排。
状态：正在检查今天的日程...
assistant：看起来 12:30 和 13:00 都可以，不会影响下午安排。你想改到哪个时间？
快捷回复：改到 12:30 / 改到 13:00 / 先不改了
```

### 展示重点

1. assistant 不是孤立聊天，而是基于来源页面工作。
2. 快捷动作等价于给 assistant 发消息。
3. 修改需要用户确认。
4. 完成后通过 Return Sync 回到日程页。

### 实现程度

```text
必须可交互：打开半屏、选择快捷回复、日程事件块更新时间
可以 mock：冲突检查结果
不用实现：复杂多事件重排、真实系统日历同步
```

## 5. Demo 3：Skill 设置与启用

### 目标

展示 Skill 是可治理的能力模块，而不是一堆拟人化 Agent。

### 链路

```text
进入角色页
→ 展示饮食助手 / News 简报 / 学习助手
→ 饮食助手状态为待设置
→ 点击“去设置”
→ 设置口味、预算、忌口、常用时间
→ 保存并启用
→ 返回角色页
→ 饮食助手状态变为已启用
```

### 设置项

```text
启用 / 停用
口味偏好：清淡 / 米饭 / 热食 / 面食
预算范围：20-50 元
忌口：少辣 / 不太油
常用时间：11:30-13:00
个性化备注：工作日午餐尽量轻一点
```

### 展示重点

1. 初始用户有内置 Skill，但不默认启用。
2. 用户需要完成必要设置后，Skill 才能参与建议。
3. Skill 只提供领域能力，最终由 assistant 调用和写回。
4. Skill 设置和全局偏好分层管理。

### 实现程度

```text
必须可交互：进入设置、选择标签、保存、状态更新
可以 mock：News / 学习助手仅展示卡片状态
不用实现：发现更多 Skill、Skill 市场、第三方安装
```

## 6. Demo 4：最近判断与 Memory 反馈

### 目标

展示系统不是一次性推荐，而是能解释、纠正和学习。

### 链路

```text
进入我的
→ 点击最近判断
→ 查看“为什么今天推荐轻食鸡胸能量碗”
→ 展示输入条件、使用偏好、当天状态、最终建议
→ 用户点击“不太准确”或输入“以后午餐别太冷”
→ 系统记录为候选偏好
→ 页面提示：已记录为候选偏好，后续会结合更多反馈调整
```

### 最近判断展示字段

```text
判断时间：今天 11:20
关联事项：午餐
输入条件：工作日午餐、当前地点、11:30-13:00 时间窗
使用偏好：清淡、高蛋白、预算 50 元以内
当天状态：下午有产品讨论，建议避免太油
最终建议：轻食鸡胸能量碗
用户反馈：以后午餐别太冷
写回状态：候选偏好，待后续验证
```

### 展示重点

1. AI 判断有依据。
2. 用户可以纠正。
3. 反馈不直接污染长期记忆，而是先进入候选。
4. 记忆增长是受治理的。

### 实现程度

```text
必须可交互：查看最近判断、提交反馈、显示候选状态
可以 mock：判断依据和写回状态
不用实现：完整 memory 检索、跨天候选提升、复杂纠偏
```

## 8. Demo 不做什么

P0 Demo 暂不做：

1. 真实外卖下单
2. 支付流程
3. 真实地图和定位
4. 真实日历双向同步
5. 复杂通知系统
6. 完整模型配置
7. 第三方 Skill 市场
8. 多 Skill 自动协作面板
9. 高风险自动代理
10. 常驻后台 agent

## 9. 验收标准

Demo 最小验收：

1. 能看到完整四 Tab：NOW / 日程 / 角色 / 我的。
2. 能看到全局 assistant 输入框。
3. 能从 NOW 确认一个默认方案。
4. 能从日程打开 assistant 半屏并修改事件时间。
5. 能设置并启用饮食助手。
6. 能查看最近判断并提交反馈。
7. 所有关键结果都写回页面，而不是只停留在聊天中。

## 10. 一句话总结

P0 Demo 要证明的不是“这个 App 功能很多”，而是证明一条 AI 生活助理主链已经成立：

```text
上下文 → 默认决策 → 用户确认 / 调整 → 页面写回 → 反馈学习
```
