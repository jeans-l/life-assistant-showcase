# 04｜Assistant Runtime：统一主循环与写回主权

> 本文件独立说明生活助理如何运行：不同入口如何进入统一主循环，Chat、行为决策和纠偏如何分工，Memory 如何读取，最终结果如何写回页面和记忆。

---

## 1. Runtime 的核心问题

生活助理不是一个“用户问一句，模型答一句”的系统。

它必须同时处理：

1. 用户自然语言输入。
2. 用户在页面里的直接操作。
3. 时间窗口触发。
4. 外部服务状态变化。
5. 通知点击后的上下文恢复。
6. 周期性的候选复核和记忆纠偏。
7. 重大任务前的证据回看。

如果这些入口各自实现一套逻辑，系统会很快失控。

所以生活助理需要一个统一 Assistant Runtime。

---

## 2. 一条总主线

Assistant Runtime 可以压缩成五段：

```text
触发
→ 路由与上下文组装
→ execution mode 核心循环
→ 正式写回
→ 更新
```

更完整地展开为：

```text
entry
→ route
→ context_load
→ domain_judgment
→ validation
→ state_recheck
→ policy_decision
→ execution_or_handoff
→ writeback_decision
→ return_sync
→ correction
```

固定原则：

1. 不是每次都走满所有阶段。
2. 阶段顺序不能反过来。
3. 写回主权只在 `writeback_decision`。
4. 页面同步只能消费已完成裁决的结果。
5. Skill、页面、Provider 都不能绕过 runtime 直接改真源。

---

## 3. 入口分类

### 3.1 六类触发

| 触发 | 含义 | 示例 |
|---|---|---|
| user_input | 用户主动输入或点快捷动作 | 帮我把午餐晚一点 |
| time_window | 时间窗口到达 | 午餐准备窗口到了 |
| external_event | 外部服务或页面编辑回流 | 外卖状态变化、用户改了日程 |
| notification_click | 用户点击通知恢复上下文 | 点击午餐提醒 |
| periodic_correction | 后台候选复核 | 昨天的临时偏好过期 |
| pre_major_task_review | 重大任务前回看 | 重要会议前复核准备事项 |

---

### 3.2 用户体感上的三类入口

对用户来说，可以理解为三类：

| 用户体感 | 系统来源 | 默认路径 |
|---|---|---|
| 用户对话 | user_input | chat mode |
| 用户直接操作 | 页面编辑产生 external_event | behavior mode 轻量影响判断 |
| assistant 自驱 | 时间窗口、外部事件、周期纠偏 | behavior mode 或 correction mode |

通知点击不是第四类产品入口，它的首要作用是恢复上下文，再回到上面三类之一。

---

## 4. 三种 Execution Mode

### 4.1 Chat Mode

Chat Mode 处理用户可见对话。

适用场景：

1. 用户输入自然语言。
2. 用户点击快捷动作。
3. assistant 自驱后需要用户确认。
4. 出现冲突，需要用户选择。
5. 用户要求解释、修改或补充。

Chat Mode 的目标不是无限聊天，而是把用户表达收敛为可执行判断包。

输出可能包括：

1. 直接回复。
2. 澄清问题。
3. 快捷选择。
4. Behavior Handoff。
5. 写回候选。
6. 失败或阻断说明。

---

### 4.2 Behavior Mode

Behavior Mode 处理围绕 item 的正式推进。

它回答：

1. 当前是否需要做这件事。
2. 是否需要调用 Skill。
3. Skill 结果是否合格。
4. 当前状态是否仍然成立。
5. 是否允许执行。
6. 是否需要用户确认。
7. 结果应如何进入写回候选。

典型场景：

1. 午餐窗口到了，assistant 主动准备方案。
2. 用户把日程从 12:00 改到 13:00，系统判断是否影响后续安排。
3. 外部服务状态变化，系统更新 item 状态。

---

### 4.3 Correction Mode

Correction Mode 处理候选复核、提升、回退和过期。

适用场景：

1. 一次反馈是否应进入长期偏好。
2. 今天状态是否自然过期。
3. 多次相同反馈是否构成稳定规律。
4. 原有长期默认是否被连续反驳。
5. 某个候选是否应该继续 hold。

Correction Mode 的目标是让系统变聪明，但不过度学习。

---

## 5. 统一阶段说明

### 5.1 Entry

回答：这次从哪里进来？

需要记录：

1. 入口类型。
2. 来源页面。
3. 来源对象。
4. 当前用户动作。
5. 是否从通知恢复。

---

### 5.2 Route

回答：进入哪种 execution mode？

规则：

1. 用户输入默认进入 Chat Mode。
2. 页面明确编辑默认进入 Behavior Mode 轻量判断。
3. 时间窗口默认进入 Behavior Mode。
4. 周期纠偏进入 Correction Mode。
5. 需要用户确认时回到 Chat Mode。

---

### 5.3 Context Load

回答：本次最小需要读哪些上下文？

默认读取顺序：

```text
trigger
→ identity
→ daily
→ interaction
→ skill-local
→ evidence gate
```

也就是：

1. 触发信息。
2. 用户长期默认。
3. 当天状态。
4. 当前对话约束。
5. Skill 本地偏好。
6. 必要时追加历史证据。

原则：

1. 不一次性塞入全量 memory。
2. 先读最小切片。
3. 历史证据只有在冲突、提升、回退、解释或纠偏时读取。

---

### 5.4 Domain Judgment

回答：是否需要领域判断，由哪个 Skill 形成方案？

例如午餐事项可能调用饮食助手。

Skill 可以返回：

1. 领域方案。
2. 领域状态。
3. 领域动作。
4. Provider 需求。
5. 写回候选。

Skill 不返回最终写回结果。

---

### 5.5 Validation

回答：当前结果是否合格？

校验维度包括：

1. 字段是否完整。
2. 时间是否有效。
3. 预算是否符合。
4. 权限是否具备。
5. 外部服务是否可用。
6. 是否存在明显冲突。

不合格时，可以补问、重算、降级或阻断。

---

### 5.6 State Recheck

回答：执行前情境是否仍成立？

例如：

1. 用户是否已经手动改了时间。
2. 当前时间是否已经错过窗口。
3. 外部候选是否过期。
4. 后续日程是否发生变化。
5. Provider 状态是否变化。

这一步防止 assistant 用过期状态执行动作。

---

### 5.7 Policy Decision

回答：当前允不允许继续推进？

结果可能是：

1. 允许直接执行。
2. 需要用户确认。
3. 需要补充信息。
4. 暂挂。
5. 阻断。
6. 降级为只解释。

---

### 5.8 Execution or Handoff

回答：现在是执行、移交、等待确认，还是只保留结果？

可能路径：

1. 页面内更新。
2. 调用 Provider。
3. 返回 Chat 等用户确认。
4. 只生成建议，不执行。
5. 记录失败原因。

---

### 5.9 Writeback Decision

回答：新增信息该进入哪里？

正式裁决结果只有四类：

| 结果 | 含义 |
|---|---|
| hold | 证据不足，继续候选态 |
| promote | 证据足够，提升到目标层 |
| rollback | 原有结论被推翻，降级或删除 |
| expire | 临时状态过期清除 |

写回对象可能包括：

1. conversationContext。
2. today。
3. skill-local。
4. decisionRecords。
5. profile 候选。
6. projection。

---

### 5.10 Return Sync

回答：哪些结果需要同步回页面？

页面同步包含：

1. 返回哪个页面。
2. 刷新哪个区域。
3. 是否保留滚动位置和日期。
4. 是否需要 toast。
5. 写回失败时如何提示。

原则：

1. 默认返回来源页。
2. 默认只刷新受影响区域。
3. 不用整页重载掩盖状态变化。
4. 结果不能只停留在聊天中。

---

### 5.11 Correction

回答：是否需要后续纠偏？

处理：

1. 临时状态过期。
2. 候选继续 hold。
3. 稳定规律 promote。
4. 被反驳默认 rollback。
5. 产品缺口进入产品候选。

---

## 6. Memory 读取策略

### 6.1 Memory 分层

| 层级 | 作用 | 示例 |
|---|---|---|
| profile | 长期稳定默认 | 常驻位置、预算、偏好 |
| assistantSoul | 表达风格 | 简洁、直接、少打扰 |
| today | 当天状态 | 天气、地点、日程、精力 |
| conversationContext | 本轮约束 | 这次想吃热一点 |
| skill-local | 领域记忆 | 饮食设置、餐史、忌口 |
| decisionRecords | 决策证据 | 当时为什么这么推荐 |
| crossSkillInsights | 跨场景规律 | 低精力日降低安排强度 |

---

### 6.2 默认读取顺序

```text
trigger
→ profile + assistantSoul
→ today
→ conversationContext
→ skill-local
→ evidence
```

原则：

1. 先长期默认。
2. 再当天状态。
3. 再当前表达。
4. 再领域偏好。
5. 最后才读证据。

---

### 6.3 Evidence Gate

历史证据不是默认读取。

只有以下情况才追加读取：

| 闸门 | 触发条件 |
|---|---|
| conflict | 当前判断和长期默认冲突 |
| promote | 候选准备升级 |
| rollback | 长期项可能被推翻 |
| review | 用户要求解释或重大任务前回看 |
| correction | 后台纠偏 |

这样能避免 memory 过载，也避免模型被大量无关历史干扰。

---

## 7. 写回策略

### 7.1 候选来源

允许产生写回候选的来源：

1. assistant。
2. skill。
3. execution result。
4. feedback。
5. page edit event。

但它们都不拥有正式写回主权。

---

### 7.2 写回判断问题

每条候选都要回答：

1. 它是事实、约束、结果、偏好还是推断？
2. 它只影响本轮、今天，还是可能长期复用？
3. 它有没有用户确认、真实结果或重复行为支撑？
4. 它写入后是否可能造成风险？
5. 它应该直接写、候选写、阻断，还是等待更多证据？

---

### 7.3 示例

#### 用户说：“这次想吃热一点”

处理：

1. 写入 conversationContext。
2. 如果用户确认今天都这样，可写入 today。
3. 不直接写入长期 profile。

---

#### 用户多次反馈：“午餐别太冷”

处理：

1. 首次进入 decisionRecords 和 skill-local 候选。
2. 多次、跨天、同上下文成立后，提升为饮食助手稳定偏好。
3. 如果后续用户否认，可以 rollback。

---

#### 外卖服务不可用

处理：

1. 不伪装成已查询。
2. 进入执行失败记录。
3. 页面展示能力受限。
4. 可提供手动处理或稍后再试。

---

## 8. Return Sync 示例

### 8.1 从 NOW 调整午餐

```text
来源页：NOW
动作：换个更轻的
assistant 调用饮食助手重新生成方案
写回裁决通过
刷新：NOW 主卡 + 日程相关事件 + 最近判断
返回：NOW
```

---

### 8.2 从日程调整时间

```text
来源页：日程
动作：帮我把午餐晚一点
assistant 判断 12:30 可行
用户确认
写回 item window
刷新：日程事件块 + NOW 当前事项
返回：日程
```

---

### 8.3 从我的修改偏好

```text
来源页：我的 / 偏好管理
动作：以后提醒少一点
assistant 生成偏好候选
用户确认保存
刷新：我的偏好摘要
后续提醒策略读取新默认
```

---

## 9. 并发与冲突原则

当多个入口同时操作同一 item：

1. 用户主动操作优先。
2. 同优先级时，后发生的操作触发轻量 replan。
3. 后台自驱不得覆盖用户刚刚完成的显式修改。
4. 冲突原因需要保留，方便后续解释和纠偏。

示例：

```text
assistant 准备把午餐安排到 12:00
用户刚手动改到 13:00
系统必须保护用户修改
assistant 进入 recheck，而不是静默覆盖
```

---

## 10. Runtime 验收标准

P0 Runtime 至少满足：

1. 任意页面进入 assistant 时能带来源上下文。
2. 快捷动作等价于带上下文的用户消息。
3. 用户直接编辑后能触发轻量影响判断。
4. Skill 结果不能直接写页面。
5. 写回必须经过统一裁决。
6. assistant 完成后能 return sync 回页面。
7. 失败、阻断、未授权要明确说明。
8. 用户中断时不写入未确认结果。
9. 反馈能进入候选或 decision record。
10. 系统不会把一次临时偏好直接写成长期默认。

---

## 11. 一句话总结

Assistant Runtime 的核心是：

> 把用户输入、页面操作、时间窗口、Skill 能力、Provider 状态和 Memory 反馈统一收敛到一条受治理的主循环里，并通过唯一写回主权把 AI 判断变成可靠的生活状态。
