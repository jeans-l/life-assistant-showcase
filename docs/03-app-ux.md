# 03｜App UX：四个页面与一个全局 Assistant

> 本文件独立说明生活助理 App 的页面结构、页面职责、核心交互和设计原则。目标是让外部读者不用看完整 PRD，也能理解产品为什么这样组织。

---

## 1. App 的定位

App 是 assistant 的服务舞台，不是第二个智能主体。

分工是：

| 对象 | 负责什么 |
|---|---|
| App 页面 | 展示、编辑、确认、承接和返回同步 |
| Assistant | 理解上下文、判断、协调、执行、写回 |
| Skill | 提供领域能力和领域结果 |
| Provider | 提供外部候选、执行和状态回流 |
| Memory | 提供长期偏好、当天状态、本轮约束和历史依据 |

一句话：

> 页面负责接住结果，assistant 负责决定结果是什么、何时出现、能否写回。

---

## 2. 主导航结构

移动端主导航固定为四项：

```text
NOW / 日程 / 角色 / 我的
```

全局 assistant 不进入底部导航，而是所有页面共享的输入与对话层。

这样设计的原因：

1. Assistant 不应被降级成一个单独页面。
2. 用户在任何场景都可能需要 assistant。
3. assistant 每次都需要带入当前页面上下文。
4. 对话结果必须回到来源页面或稳定父页面。

---

## 3. 页面职责总览

| 页面 | 定位 | 回答的问题 |
|---|---|---|
| NOW | 当前注意力收敛页 | 现在最该关注什么 |
| 日程 | 完整时间组织页 | 今天和接下来怎么排 |
| 角色 | Skill 管理页 | 哪些能力可用，哪些需要设置 |
| 我的 | 长期默认与控制权中心 | 系统记住了什么，我能怎么改 |
| 全局 Assistant | 判断、协调、执行和写回层 | 复杂请求如何继续处理 |

---

## 4. NOW

### 4.1 页面目标

NOW 只回答三件事：

1. 当前整体状态怎样。
2. 现在最该接住哪一件事。
3. 此刻最值得做的一个动作是什么。

NOW 不是完整时间表，不展示全天日程列表。

---

### 4.2 首屏结构

有当前事项时：

1. 顶部日期与状态区。
2. 当前事项主卡。
3. 快捷动作区。
4. 全局 assistant 输入框。
5. 底部主导航。

无当前事项时：

1. 顶部日期与状态区。
2. 空态主卡。
3. 轻提示卡。
4. 全局 assistant 输入框。
5. 底部主导航。

---

### 4.3 顶部日期与状态区

展示内容：

1. 日期：2026 年 04 月 27 日。
2. 星期：周一。
3. 地点：上海静安寺。
4. 天气：22°C。
5. 简短建议：建议穿薄外套，早晚温差约 5°C。

规则：

1. 顶部只做状态摘要。
2. 天气不是独立大卡片。
3. 地点可点击切换或授权。
4. 天气和地点不可用时，不阻断主流程。

---

### 4.4 当前事项主卡

主卡展示：

1. 状态标签：准备中 / 进行中 / 已确认。
2. 事项标题：午餐。
3. 时间窗：11:30 - 13:00。
4. 判断说明：现在适合先把午餐定下来，避免 12 点后排队。
5. 领域结果卡：轻食鸡胸能量碗。
6. 主动作：确认。
7. 次动作：调整。
8. 更多入口。

事项和方案必须分开：

```text
事项：午餐
方案：轻食鸡胸能量碗
```

这样后续才能做到：换方案不等于删除事项，改时间不等于改餐食偏好。

---

### 4.5 NOW 快捷动作

示例：

1. 换个更轻的。
2. 更快送达。
3. 晚点提醒我。

规则：

1. 点击快捷动作不是前端直接改卡片。
2. 它等价于向 assistant 发送一条带上下文的消息。
3. assistant 判断后刷新当前主卡或相关日程。

---

### 4.6 NOW 空态

文案建议：

```text
有什么需要安排吗？
你可以直接告诉我想做什么，也可以先看看今天的时间。
```

主按钮：`让我帮你安排一下`。

规则：

1. 不强制用户添加 Skill。
2. 不强制用户创建日程。
3. 用户可以直接唤起 assistant。

---

## 5. 日程

### 5.1 页面目标

日程负责完整时间结构。

它回答：

1. 当前日期有哪些安排。
2. 哪些时间段被占用。
3. 哪些时间段空着。
4. 某个事件的详情和调整入口在哪里。

---

### 5.2 页面结构

移动端结构：

1. 顶部年月。
2. 搜索按钮。
3. 新建日程按钮。
4. 周日期条。
5. 全天区。
6. 时间栅格。
7. 当前时间线。
8. 事件块。
9. 快捷动作：新建日程 / 调整今天 / 找空档。
10. 全局 assistant 输入框。
11. 底部主导航。

---

### 5.3 事件块

事件块展示：

1. 标题。
2. 时间。
3. 地点。
4. 状态或来源标记。

示例：

```text
午餐
11:30 - 13:00
楼下轻食店
```

规则：

1. 点击事件块进入事件详情。
2. 不使用“午餐决策”这类系统过程词。
3. 标题使用用户可理解文案。
4. 当前时间线不遮挡主体信息。

---

### 5.4 事件详情页

事件详情页用于查看完整信息，不直接编辑复杂字段。

结构：

1. 顶部导航：返回、事件详情、更多。
2. 主信息卡：标题、状态、时间、地点、来源。
3. 推荐方案卡：Skill 结果。
4. 处理建议卡：晚点提醒、换个更快的。
5. 备注卡：目标、约束、补充说明。
6. 全局 assistant 输入框。

更多菜单包含：

1. 编辑日程。
2. 复制日程。
3. 删除日程。

---

### 5.5 新建日程页

进入方式：

1. 顶部新建按钮。
2. 底部快捷按钮。
3. 点击空白时间段。
4. assistant 创建后需要用户补充字段。

最少字段：

1. 事项标题。
2. 日期时间。
3. 地点，可选。
4. 提醒，可选。
5. 重复，可选。
6. 分类，可选。
7. 备注，可选。

规则：

1. 标题 + 日期时间是最小保存条件。
2. 从空白时间段进入时自动预填时间。
3. 单事件明确字段直接保存。
4. 多事件重排和模糊目标进入 assistant。

---

### 5.6 编辑日程页

编辑页用于明确字段修改。

结构：

1. 基础信息：标题、时间、地点。
2. 事项内容：当前推荐或领域结果。
3. 处理建议：调整餐食、晚点提醒。
4. 备注：目标、约束、补充说明。
5. 影响提示：修改会先保存，如影响其他安排，我会再提醒你。

规则：

1. 单事件字段修改先页面保存。
2. 保存后同步给 assistant 做轻量影响判断。
3. 如果影响多个事项，再进入 assistant。
4. 用户没有主动进入聊天时，不强制打开聊天。

---

### 5.7 删除确认

删除必须二次确认。

底部 sheet 包含：

1. 标题：删除日程？
2. 说明：删除后将从日程中移除“午餐”。
3. 事件摘要。
4. 主危险按钮：删除。
5. 次按钮：返回。

规则：

1. 删除成功后返回时间栅格。
2. 不重复弹成功 toast。
3. 外部只读事件不提供假删除。

---

## 6. 角色

### 6.1 页面目标

角色是 Skill 的用户可理解包装。

它负责：

1. 展示内置 Skill。
2. 展示状态。
3. 查看详情。
4. 进入设置。
5. 启用或暂停。

---

### 6.2 P0 Skill 状态

P0 展示三个内置 Skill：

| Skill | 状态 | 主按钮 |
|---|---|---|
| 饮食助手 | 待设置 | 去设置 |
| News 简报 | 待设置 | 去设置 |
| 学习助手 | 已停用 / 待设置 | 启用 / 去设置 |

初始不默认启用任何 Skill。

---

### 6.3 Skill 状态含义

| 状态 | 含义 |
|---|---|
| 待设置 | 需要补充必要配置后才能启用 |
| 已启用 | assistant 可以在合适时机调用 |
| 已停用 | 不主动参与建议，但历史保留 |

---

### 6.4 Skill 详情页

详情页解释这个 Skill 能做什么，不负责复杂编辑。

结构：

1. Skill 概览。
2. 能帮什么。
3. 常用偏好摘要。
4. 立即使用。
5. 偏好设置。
6. 快捷入口。

规则：

1. 立即使用会进入 assistant。
2. 待设置时先引导完成设置。
3. 已停用时先提示启用。

---

### 6.5 Skill 设置页

以饮食助手为 P0 标准。

字段：

1. 启用 / 停用。
2. 口味偏好。
3. 预算范围。
4. 忌口。
5. 常用时间。
6. 个性化备注。

保存规则：

1. 必要字段完整且开关开启，状态变为已启用。
2. 开关关闭，状态变为已停用。
3. 必要字段缺失，保持待设置。
4. 修改只影响该 Skill，不修改全局偏好。

---

## 7. 我的

### 7.1 页面目标

我的是长期默认、偏好、记忆、依据和控制权中心。

它回答：

1. 系统当前怎么看我。
2. 我接入了哪些服务。
3. 系统记住了什么。
4. 我可以改哪些默认处理方式。
5. 最近一次判断依据在哪里看。

---

### 7.2 默认页结构

1. 顶部标题：我的。
2. 用户摘要卡。
3. 助手风格卡。
4. 基础信息。
5. 偏好管理。
6. 记忆与习惯。
7. 最近判断。
8. 最近聊天。
9. 模型服务 / 已接入服务。
10. 快捷指令。
11. 全局 assistant 输入框。

---

### 7.3 最近判断

最近判断需要展示：

1. 判断时间。
2. 关联事项。
3. 输入条件。
4. 使用的偏好或约束。
5. 最终建议。
6. 用户是否采纳。
7. 用户反馈入口。

原则：

1. 解释要短。
2. 不写模型技术细节。
3. 用户可以纠正。
4. 反馈进入记忆候选或偏好修正链路。

---

### 7.4 已接入服务

P0 最少展示：

1. 模型服务。
2. 外卖服务商。
3. 后续可扩展日历、定位、提醒、麦克风。

状态包括：

1. 已接入。
2. 未接入。
3. 异常。
4. 需授权。

原则：

1. 外部服务未接入时，不能伪装真实调用。
2. 模型未接入时，需要说明复杂判断能力受限。
3. 服务状态变化后，刷新对应页面能力边界。

---

## 8. 全局 Assistant

### 8.1 输入框状态

所有主页面默认展示全局 assistant 输入框。

结构：

1. assistant 头像。
2. 输入区域。
3. 语音按钮。
4. 发送按钮。
5. 可选快捷短句。

点击输入框后进入半屏 conversation。

---

### 8.2 半屏 conversation

半屏用于让用户感知 assistant 仍基于当前页面工作。

结构：

1. 背景页面虚化或暗化。
2. 底部弹层。
3. 拖拽柄。
4. assistant 头像。
5. 标题。
6. 副标题。
7. 上下文条。
8. 消息区。
9. 快捷选项。
10. 输入框。

上下文条示例：

```text
当前页面：NOW
已带入：4月27日 · 午餐 · 11:30-13:00
```

---

### 8.3 全屏 conversation

全屏用于长对话和复杂任务。

规则：

1. 全屏不是第五个 Tab。
2. 全屏仍保留上下文条。
3. 关闭后返回来源页或稳定父页面。
4. 完成后必须写回页面。

---

## 9. 共享交互规则

### 9.1 页面状态

所有页面共享五种状态：

| 状态 | 作用 |
|---|---|
| collapsed | 默认承接态 |
| detail | 查看完整信息 |
| editor | 字段编辑 |
| sheet | 一次确认或轻量选择 |
| conversation | assistant 对话层 |

---

### 9.2 什么时候页面直接处理

页面可以直接完成：

1. 新建一个明确日程。
2. 编辑单事件字段。
3. 删除单事件。
4. 保存 Skill 设置。
5. 修改基础偏好字段。

---

### 9.3 什么时候必须进入 assistant

以下情况必须进入 assistant：

1. 用户用自然语言表达目标。
2. 点击快捷动作。
3. 一次修改影响多个事项。
4. 需要外部服务。
5. 需要跨页面协调。
6. 需要判断是否写入 memory。
7. 需要解释最近判断。

---

## 10. 体验验收标准

P0 App 体验需要满足：

1. 用户一眼知道 NOW 当前要关注什么。
2. 日程页和 NOW 不重复。
3. Skill 状态清晰，不默认越权。
4. 我的页能体现控制权和可解释性。
5. assistant 半屏能明显感知来源页面。
6. 结果不会只停留在聊天里。
7. 页面、输入框、底部导航不互相遮挡。
8. 空态不是失败状态。
9. 未接入服务不伪装已完成。
10. 用户随时可以修改、关闭或纠正。

---

## 11. 一句话总结

App UX 的核心是：

> 用四个页面承接生活场景，用全局 assistant 处理判断和写回，让用户始终感觉自己在操作生活，而不是在和一个孤立聊天框对话。
