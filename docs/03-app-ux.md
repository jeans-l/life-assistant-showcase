# 03｜App UX Showcase

> 这份文档用于说明 GitHub showcase 中如何展示当前移动端设计稿，以及每个页面应该表达的产品含义。

## 1. 视觉方向

整体视觉采用轻量、可信、低打扰的移动端卡片风格。

关键词：

```text
浅色背景
大圆角卡片
低饱和图标
绿色主动作
清晰层级
轻量阴影
底部全局 assistant 输入框
```

产品不应该像普通日程工具，也不应该像纯聊天 App。视觉上要让用户感受到：这是一个能承接生活状态、做当前判断、又不会过度打扰的个人助理。

## 2. 页面总览

移动端主导航固定为：

```text
NOW / 日程 / 角色 / 我的
```

全局 assistant 输入框位于内容区下方、底部导航上方。

```text
页面负责承接
assistant 负责判断
Skill 负责领域能力
Memory 负责持续理解
```

## 3. NOW

![NOW](../assets/screenshots/首页-1.png)

NOW 是当前注意力收敛页。

它不展示完整日程，也不做信息流，而是回答：

> 现在最该关注什么？

页面结构：

1. 顶部日期、星期、地点、天气
2. 当前事项主卡
3. 推荐方案卡
4. 主动作 / 次动作
5. 快捷动作
6. 全局 assistant 输入框
7. 底部导航

核心体验：

```text
系统在合适时间把“午餐”浮到前台
给出一个默认方案
用户可以确认，也可以调整
```

## 4. 日程

![日程](../assets/screenshots/日程-1.png)

日程是完整时间组织页。

它回答：

> 今天和接下来怎么排？

页面结构：

1. 顶部年月和操作入口
2. 日期条
3. 全天区
4. 时间栅格
5. 当前时间线
6. 事件块
7. 快捷动作
8. 全局 assistant 输入框
9. 底部导航

核心体验：

```text
明确字段编辑可以在页面内完成
复杂调整进入 assistant
assistant 完成后写回日程事件块
```

## 5. 角色

![角色](../assets/screenshots/角色-1.png)

角色页是 Skill 的用户可理解包装。

它回答：

> 哪些能力可用，哪些还需要设置？

页面结构：

1. 标题与说明
2. Skill 卡片列表
3. 状态标签
4. 主操作：去设置 / 启用 / 暂停
5. 查看详情入口
6. 全局 assistant 输入框
7. 底部导航

核心体验：

```text
Skill 是能力模块
用户可以启用、暂停、设置
Skill 不直接执行全局写回
```

P0 初始不默认启用任何 Skill。

## 6. 我的

![我的](../assets/screenshots/我的-1.png)

我的页是长期默认、偏好、记忆、依据和控制权中心。

它回答：

> 系统怎么看我？系统记住了什么？我可以怎么改？

页面结构：

1. 用户摘要卡
2. 助手风格
3. 基础信息
4. 偏好管理
5. 记忆与习惯
6. 最近判断
7. 最近聊天
8. 模型服务 / 已接入服务
9. 快捷指令
10. 全局 assistant 输入框
11. 底部导航

核心体验：

```text
用户始终保留控制权
系统判断有依据
偏好和记忆可以查看、修改、删除
```

## 7. Chat 半屏

![Chat 半屏](../assets/screenshots/chat-半屏.png)

assistant 半屏用于在不脱离当前页面的情况下，处理自然语言请求、快捷动作和复杂调整。

页面结构：

1. 背景页面保留并虚化
2. 底部半屏弹层
3. 拖拽柄
4. assistant 头像、标题、副标题
5. 上下文条
6. 消息区
7. 快捷回复
8. 输入框

核心体验：

```text
用户能感知 assistant 正基于当前页面工作
assistant 不取代页面
任务完成后结果回到来源页
```

P0 聊天只做纯文本、执行状态、快捷回复和结果摘要，不做复杂卡片表单。

## 8. 关键交互原则

### 8.1 assistant 不是第五个 Tab

assistant 是全局能力层，不进入底部导航。

### 8.2 快捷动作等价于给 assistant 发消息

例如点击“换个更轻的”，实际进入 assistant 的消息是：

```text
帮我把午餐换得更轻一点
```

前端不直接原地改结果。

### 8.3 页面承接结果，assistant 负责判断

页面负责展示、编辑、确认；assistant 负责复杂判断、跨页协调、外部服务调用和写回。

### 8.4 结果不能只停留在聊天里

assistant 处理完成后，必须写回来源页或稳定父页面。

### 8.5 空态不是异常

无当前事项、无启用 Skill、无日程，都应该是正式状态，不应强迫用户先配置。

## 9. 展示建议

GitHub README 第一屏建议放 5 张图：

```text
NOW
日程
角色
我的
Chat 半屏
```

它们刚好表达完整产品心智：

```text
当前关注 → 时间组织 → 能力管理 → 个人控制权 → 全局助理
```

## 10. 一句话总结

这套 UI 的重点不是好看，而是证明一个产品结构成立：

> App 是 assistant 的舞台，页面承接结果，assistant 完成判断，Skill 提供能力，Memory 支撑持续理解。
